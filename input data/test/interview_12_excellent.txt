INTERVIEWER:
Let’s dive into some technical depth. One of the big challenges we’re facing is around noisy neighbor problems in our Kubernetes clusters. We’ve got a mix of stateless services and stateful workloads like Kafka and databases running on the same clusters, and we’re seeing latency spikes in our critical path services when batch jobs or lower-priority workloads consume too many resources. How would you approach diagnosing and mitigating this?

CANDIDATE:
This is a classic problem, and I’ve dealt with it in a few different ways at Stripe. The first step is always observability—you can’t fix what you can’t measure. We’d want to instrument the clusters with tools like Prometheus to track CPU, memory, disk I/O, and network bandwidth at the pod, node, and namespace levels. Specifically, I’d look for metrics like:

- CPU throttling events (`container_cpu_cfs_throttled_seconds_total`)
- Memory pressure (`container_memory_working_set_bytes` vs. `container_memory_limit_bytes`)
- Disk I/O latency and throughput (`container_fs_reads_total`, `container_fs_writes_total`)
- Network bandwidth (`container_network_receive_bytes_total`, `container_network_transmit_bytes_total`)

For stateful workloads like Kafka, we’d also want to monitor disk queue depth and fsync latency, as those can be early indicators of noisy neighbors.

Once we have the data, the next step is to establish baselines and set up alerts for when resource usage deviates from expected patterns. For example, if a batch job is supposed to run at 2 AM and only use 50% of a node’s CPU, but we see it spiking to 90% and causing latency in a co-located critical service, that’s a clear signal.

For mitigation, there are a few approaches I’d consider, depending on the workload:

1. **Resource Requests and Limits**: This is the most straightforward fix. We’d work with the teams owning the noisy workloads to set appropriate requests and limits. For example, if a batch job is causing CPU throttling, we might set a CPU limit of 2 cores and a request of 1 core to ensure it doesn’t starve other workloads.

2. **Priority Classes and Preemption**: Kubernetes has a priority class system that allows you to define which pods should be scheduled first and which can be preempted if resources are scarce. We could define a `critical` priority class for latency-sensitive services and a `batch` priority class for lower-priority workloads. This ensures that critical pods get scheduled first and can preempt batch pods if needed.

3. **Node Affinity and Taints/Tolerations**: For workloads that are particularly sensitive to noisy neighbors, we could use node affinity to schedule them on dedicated nodes. For example, we might taint a set of nodes with `workload=critical` and only allow pods with the corresponding toleration to run on them. This is more of a brute-force approach, but it can be effective for workloads that are extremely sensitive to resource contention.

4. **Vertical Pod Autoscaler (VPA)**: For workloads where it’s hard to predict the right resource requests, VPA can dynamically adjust requests and limits based on historical usage. This is especially useful for stateful workloads where resource needs might change over time.

5. **Cluster Autoscaler with Overprovisioning**: If we’re seeing frequent resource contention, we might need to scale up the cluster. We could use the Cluster Autoscaler to add nodes when resources are scarce, and we could also overprovision the cluster with low-priority pods that can be evicted when higher-priority workloads need the resources.

At Stripe, we ended up using a combination of these approaches. For example, we used priority classes for our payment processing services and taints/tolerations for our Kafka clusters. We also built a tool that automatically adjusted resource requests for batch jobs based on their historical usage patterns, which reduced manual overhead for the teams owning those jobs.
